Help Desk AI Toolkit

Two LLM-powered tools for an IT help desk, built on a shared classification core: an autonomous Tier-1 support agent that resolves routine issues with the user and escalates the rest to a technician, and a portable bulk categorizer that cleans up a backlog of mis-categorized tickets. The tools are tied together by a hybrid model architecture exposed as a single configuration boundary — the design decision the whole project is built to demonstrate.

The Problem

Inbound support tickets arrive with wildly uneven detail. Technicians spend their first contact gathering basics — device, error message, what the user already tried — before any real work starts. Categorization and priority are applied inconsistently at intake, which makes routing, reporting, and trend analysis unreliable. Separately, historical tickets accumulate wrong, missing, or placeholder categories, so the one field used for reporting and routing can't be trusted.

The toolkit attacks both problems: resolve what can be resolved at first contact for new tickets, and clean up the category data on old ones. The agent handles routine issues with the user directly and escalates the rest to a technician with full context.

Tool 1 — Tier-1 Support Agent

An autonomous agent that tries to resolve the ticket with the user, and escalates cleanly when it can't. Two-stage routing classifies the ticket and scores confidence: a high-confidence routine issue gets a canned, knowledge-base-backed self-service answer with no model call at all; everything else enters a conversational path where the agent retrieves KB context and walks the user through a fix one step at a time.

If the user confirms it's fixed within a few turns, it's resolved — no ticket. If not, the agent creates a categorized ticket carrying the priority and the full transcript, and routes it to the help desk queue so a technician starts with context instead of a blank page. The conversation is deliberately bounded — it works the issue for a few turns, then hands off rather than looping forever.

Tool 2 — Bulk Categorizer

A portable, config-driven utility that reads a CSV of tickets and assigns a clean category to each in two stages. A free, offline keyword pass resolves the obvious tickets; only the genuinely ambiguous ones are sent to a model. A dry-run mode shows exactly what would change before anything is written. On the bundled synthetic backlog, the keyword stage alone resolves roughly 85% of tickets before any model is called.

The Architecture — Hybrid, as One Config Switch

Both tools sit on the same model client, which exposes one interface over three interchangeable backends. Which one runs is a single environment variable — no other code changes.

Deterministic
Offline, zero-setup default. Real implementation of the interface, so the whole pipeline runs with no model installed.
Local model
An internally-hosted open-source model. The cheap, high-volume runtime tier — no ticket data leaves your infrastructure.
Frontier model
A top-tier hosted model for the genuinely hard reasoning — the "architect" tier, used where it earns its cost.

Why this is the point

Cost scales with difficulty, not volume. Routine tickets are handled by a keyword pass or a cheap local model; only ambiguous cases reach a frontier model.

Privacy by default. Point the switch at a local model and ticket text — which routinely contains names, asset tags, and account details — never leaves the building. The frontier tier is opt-in, for the work that benefits from it.

Frontier as architect, local as runtime. A frontier model is ideal for design and supervising edge cases; a small local model is ideal for the thousand routine inferences a day. Making the boundary one config line means you choose per deployment, not per rewrite.

Security Posture
  • Bounded autonomy. The agent's actions are limited to talking to the user and opening an intake ticket; it escalates anything it can't resolve and takes no destructive or privileged actions on its own.
  • Model output is untrusted data. Every model turn passes through a validation layer that constrains the status to a known set and guards the message before the agent loop acts on it — a malformed or surprising response can't drive the loop off the rails.
  • Synthetic data only. The bundled tickets and knowledge base are generated and seeded; there is no resemblance to any real organization's data.
Try It / Read It

The repository is self-contained and runs with zero setup on the default provider — clone it, generate the synthetic data, and run either tool in three commands. The code is written to be read: legible, commented, and structured so the architecture's selling point is visible rather than buried.
